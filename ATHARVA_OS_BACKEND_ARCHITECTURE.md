# BACKEND ARCHITECTURE & API SPECIFICATION
## PRODUCT: ATHARVA OS (iOS Native Application)
### Version: 1.0.0
### Date: July 21, 2026

---

## 1. System Architecture

ATHARVA OS uses a **hybrid synchronization architecture**. The client application reads and writes data locally using SwiftData and syncs directly with **iCloud CloudKit** record zones for personal, zero-knowledge user storage. 

To support operations that cannot run natively on-device, the client communicates with a cloud backend via a secure API Gateway.

```
                  [ CLIENT APPLICATION (iOS SwiftData) ]
                   │                                  │
                   ▼ (Sync Zone Datasets)             ▼ (Core Service Requests)
      [ iCloud CloudKit Zones ]                  [ Envoy API Gateway ]
                                                      │
         ┌────────────────── Microservice Router ─────┼──────────────────┐
         ▼                                            ▼                  ▼
[ Core Sync Service ]                       [ Compiler Sandbox ]   [ AI Board Executor ]
  - PostgreSQL / Redis                        - Isolated pods        - LLM orchestration
```

---

## 2. Backend Technology Stack Recommendation

* **API Gateway & Routing**: **Envoy Proxy** or **Kong API Gateway** (handles SSL termination, CORS checking, and rate-limiting).
* **Application Services Runtime**: **Go (Golang)** or **Node.js with TypeScript** (chosen for concurrent request handling and fast execution).
* **Primary Database (Cloud Engine Backup)**: **PostgreSQL 16** with replication lag configurations.
* **In-Memory Caching & Session Management**: **Redis 7** (for API limit tracking, tokens blacklisting, and active socket registers).
* **Task Queuing & Async Pipelines**: **RabbitMQ** or **Apache Kafka** (distributes code sandboxes, PDF report compiling, and email logs).
* **Sandbox Execution Layer**: **Docker containers** managed via Kubernetes, using **gVisor** runtimes to isolate code compilation processes.

---

## 3. API Architecture

* **REST Paradigm**: Standard endpoints use RESTful structures. Payloads are formatted as JSON, utilizing camelCase property naming.
* **HTTP Verbs**:
  * `GET`: Query records (read operations).
  * `POST`: Create entities, initiate calculations, or trigger synchronizations.
  * `PUT`: Replace resource attributes.
  * `PATCH`: Partially modify fields.
  * `DELETE`: Set soft-delete flags on resources.
* **GraphQL Transition Readiness**: 
  * The internal service layer separates controller logic from database operations using the Repository Pattern. This design allows you to add a GraphQL schema layer in the future without changing the underlying business logic.

---

## 4. Authentication Strategy

* **Flow Protocol**: Authentication relies on the **OAuth 2.0 Authorization Code Flow with PKCE** (Proof Key for Code Exchange) to secure mobile client token exchanges.
* **Biometric Trust Signatures**:
  * Following first authorization, the client registers a public key generated on the device's Secure Enclave with the backend.
  * Future interactions (such as vault access or financial balance updates) use a cryptographic signature generated by the device's Secure Enclave to verify identity, eliminating the need to re-transmit the user's password.

---

## 5. Authorization & Role-Based Access Control (RBAC)

Access permissions are enforced at the API Gateway using JWT claims:
* **Tiers & Scopes**:
  * `scope:core_read`: Read access to basic metrics (habits, sleep, tasks).
  * `scope:sandbox_execute`: Permission to compile Java/Python snippets.
  * `scope:executive_access`: Full permission for financial records, cap tables, and AI Board chats.
* **RBAC Context checks**: The gateway verifies that the user ID in the resource path matches the token's subject claim (`sub`), preventing cross-tenant access.

---

## 6. Session Management

* **Stateless Token Management**: User sessions are stateless. Tokens are signed cryptographically to verify authenticity.
* **Redis Verification Registry**:
  * Active sessions are tracked in a Redis cluster:
    * Key: `session:{userId}:{deviceId}` | Value: `tokenFamilyUUID`
  * Allows immediate, global session revocation if a device is reported lost or a security anomaly is detected.

---

## 7. Token Management

* **Access Token**: JSON Web Token (JWT) signed with RS256 using private keys.
  * Expiry: $15\text{ minutes}$.
  * Payload includes user scopes and active transformation stages.
* **Refresh Token**: Randomly generated 256-bit cryptographically secure string.
  * Expiry: $30\text{ days}$.
  * Managed via strict **Refresh Token Rotation (RTR)** rules:
    * Every token refresh invalidates the old token and returns a new pair.
    * If a refresh token is used twice, the backend assumes a theft attempt, invalidates the entire token family, and prompts a password reset.

---

## 8. API Versioning Strategy

* **URL Routing Path**: The API version is prepended to the URL path:
  * `https://api.atharvaos.com/v1/sync`
* **Deprecation Policy**:
  * Deprecated versions include an HTTP header: `Warning: 299 - "API version v1 is deprecated and will be removed on YYYY-MM-DD"`.

---

## 9. Error Handling Standards

All API errors return a standard JSON payload with a `4xx` or `5xx` HTTP status code:

```json
{
  "errorCode": "ERR_COMPILATION_TIMEOUT",
  "message": "The compiler execution exceeded the 3.0 second sandbox limit.",
  "timestamp": "2026-07-21T17:00:00.000Z",
  "details": [
    {
      "field": "submittedCode",
      "issue": "Infinite loop suspected in thread allocations."
    }
  ]
}
```

---

## 10. Validation Standards

* **Content Validation (JSON Schema)**: Request payloads are validated against strict JSON schemas before being passed to business controllers.
* **Constraints**:
  * Text inputs: Stripped of HTML tags to prevent cross-site scripting (XSS).
  * Request sizes: The API Gateway limits request payloads:
    * Standard inputs: Maximum $256\text{ KB}$.
    * Media uploads (voice notes): Maximum $10\text{ MB}$.

---

## 11. Logging Strategy

* **Structured Logging**: Application logs are formatted as JSON and written to standard output.
* **Distributed Tracing**:
  * Every request is stamped with a unique `X-Correlation-ID` header at the API Gateway.
  * This correlation ID is passed to all internal services and database query logs, allowing tracing of actions in Envoy, Go microservices, and PostgreSQL.

---

## 12. Monitoring Strategy

* **Ingestion Pipelines**: Services expose metrics at `/metrics` for Prometheus scraping.
* **Primary SLO KPIs**:
  * **Availability**: $> 99.95\%$ success rates on all core routes.
  * **Latency**: P95 latency $< 150\text{ms}$ for standard sync operations; p95 latency $< 2.5\text{s}$ for compiler executions.
  * **Errors**: Trigger warning events if API error rates exceed $1\%$ within a 5-minute window.

---

## 13. Rate Limiting

Rate limiting is enforced at the API Gateway based on the token identifier:
* **Standard Tier**: Capped at 100 requests per minute.
* **Executive Subscription Tier**: Capped at 1,000 requests per minute.
* **AI Engine Consultations**: Capped at 20 remote prompts per hour to control LLM costs.

---

## 14. Security Best Practices

* **CORS Policies**: Restricted to Apple application network origins.
* **Secure Transport**: All connections require HTTPS using TLS 1.3. Pre-flight checks drop connection requests running TLS 1.1 or 1.2.
* **OWASP Top 10 Protections**:
  * Prevent SQL injection through database parameter binding.
  * Prevent XML entity injection by disabling DTD parsers.
  * Sanitize inputs to prevent Cross-Site Scripting (XSS).

---

## 15. Encryption Strategy

* **Transit Security**: TLS 1.3 encryption for all data in transit.
* **Field-Level Envelope Encryption**:
  * Sensitive properties (financial entries, journals) are encrypted client-side using AES-GCM-256 before being sent to the API.
  * The cloud database stores only raw, encrypted byte blocks, ensuring the cloud database administrator cannot read personal user data.

---

## 16. File Upload & Storage Architecture

* **Storage Orchestration**:
  * Large documents and voice note recordings are stored in secure cloud object stores (e.g., AWS S3).
* **Upload Pipeline**:
  1. Client sends a request to the backend for a pre-signed upload URL.
  2. Backend returns a temporary, restricted URL (`PUT /uploads/{hash}`) valid for $5\text{ minutes}$.
  3. Client uploads the media file directly to S3, bypassing application servers.
  4. S3 triggers an asynchronous scan task using an antivirus scanner before marking the asset reference as valid.

---

## 17. Notification Service Architecture

* **APNS Integration**:
  * The notification service connects directly to the Apple Push Notification Service (APNS) using HTTP/2 protocols.
  * Employs token-based authentication with Apple Developer signing keys.
* **Queue Strategy**:
  * Low-priority alerts (weekly reports) are batched and scheduled for delivery during the user's active evening hours.
  * High-priority discipline alerts bypass batches for immediate delivery.

---

## 18. AI Service Integration Architecture

* **Orchestration Gateway**:
  * Connects to external LLM services (GPT-4o, Gemini 1.5 Pro) via a secure internal proxy.
* **Context Assembly**:
  * The proxy retrieves the user's active stage details, recent journal sentiment vectors, and active runways. It compiles these metrics into a system prompt, formatting the request as context for the advisory board members (CTO, CFO, CMO, Coach).
* **Streaming Tokens Protocol**:
  * Uses Server-Sent Events (SSE) to stream advisor responses to the client, keeping first-token latency under $800\text{ms}$.

---

## 19. Analytics Pipeline

* **Data Ingestion**:
  * Metrics are queued asynchronously using **Kafka** to prevent locking core sync APIs.
* **Columnar Datastore**:
  * Data points are ingested into a **ClickHouse** columnar database. ClickHouse runs queries on performance logs across users to optimize roadmap designs.

---

## 20. Backup & Recovery Strategy

* **Database Backups**:
  * Daily snapshots of the PostgreSQL database, combined with Write-Ahead Logging (WAL) archiving every 15 minutes to secure S3 storage.
* **Recovery SLAs**:
  * Recovery Point Objective (RPO): $< 15\text{ minutes}$ of potential data loss.
  * Recovery Time Objective (RTO): $< 2\text{ hours}$ to restore full platform services.

---

## 21. Cloud Synchronization Strategy

The sync engine uses incremental change tokens to manage data transfers:
* **Upload Diff Payload**: The client packages local updates into a JSON object containing the target table name, record UUID, vector clock, and encrypted properties.
* **Vector Check Engine**: The server evaluates the incoming vector clocks against the cloud database. It writes changes to PostgreSQL, updates the cloud change token, and returns any modifications made by other devices.

---

## 22. Offline Synchronization Strategy

* **Sync Status Pipeline**:
  * When a device reconnects after operating offline, it sends a payload to `POST /v1/sync` containing its active change tokens and queued updates.
* **Conflict Resolution**:
  * The backend processes updates sequentially. If a conflict occurs, it applies the vector clock resolution rules (preserving the higher clock value) and returns the outcome in the response payload to allow the client to resolve the collision locally.

---

## 23. Scalability Strategy

* **Database Sharding**:
  * PostgreSQL uses horizontal sharding partitioned by `user_id` range hashes. This isolates user data to specific database instances, allowing performance to scale linearly as the user base grows.
* **Stateless Service Scale**:
  * Go/TypeScript microservices run inside Kubernetes pods configured to scale automatically (Horizontal Pod Autoscaler) when CPU utilization exceeds $70\%$.

---

## 24. Performance Optimization Strategy

* **Database Connection Pooling**:
  * Uses **PgBouncer** to pool PostgreSQL connections, reducing latency overhead for database writes during synchronizations.
* **Read Cache**:
  * Active user configuration states and roadmap structures are cached in Redis to minimize database read overhead.

---

## 25. Caching Strategy

* **Invalidation Triggers**:
  * Dynamic cache states are invalidated when a `POST` or `PATCH` modification occurs on the resource path.
* **Cache Eviction**:
  * Uses a Least Recently Used (LRU) policy.
  * TTL configurations:
    * Static roadmaps: 24 hours.
    * Active user profiles: 1 hour.
    * Revoked token blacklists: 15 minutes (matching token lifetimes).

---

## 26. API Documentation Standards

* **OpenAPI 3.0 Standard**: All endpoints are documented using OpenAPI 3.0 YAML specifications.
* **Schema Generation**:
  * Documentation is generated directly from code comments, ensuring the API documentation matches the live backend routes.

---

## 27. Third-Party Integration Strategy

* **Plaid API Integration**:
  * Integrates with Plaid to retrieve bank balances. The Plaid server triggers webhook notifications (`POST /v1/integrations/plaid/webhook`) when new transactions are reconciled, updating the user's savings metrics.
* **Apple HealthKit & EventKit**:
  * Client devices extract sleep, workout, and calendar metrics locally and upload changes during synchronization sweeps, keeping the backend isolated from direct Apple system APIs.

## 28. Core API Schemas

### 28.1 Authentication APIs

#### 28.1.1 POST /v1/auth/signup
* **Purpose**: Registers a new user account.
* **Authentication Required**: No.
* **Request Header**: `Content-Type: application/json`
* **Request Body**:
```json
{
  "email": "rohan@domain.com",
  "password": "Password123!",
  "displayName": "Rohan Dev"
}
```
* **Validation Rules**:
  * `email`: Must match a standard email regex structure.
  * `password`: Minimum 8 characters, must include 1 uppercase letter, 1 lowercase letter, and 1 special symbol.
* **Success Response (201 Created)**:
```json
{
  "message": "User registered successfully. Verification email sent.",
  "userId": "7a9f8b4d-1234-4bc3-90ef-56ad98ef7cf1"
}
```
* **Error Response (400 Bad Request)**:
```json
{
  "errorCode": "ERR_EMAIL_EXISTS",
  "message": "The email address is already registered."
}
```

#### 28.1.2 POST /v1/auth/login
* **Purpose**: Authenticates credentials and returns token keys.
* **Authentication Required**: No.
* **Request Body**:
```json
{
  "email": "rohan@domain.com",
  "password": "Password123!"
}
```
* **Success Response (200 OK)**:
```json
{
  "accessToken": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiI3YTlmOGI0ZC0xM...",
  "refreshToken": "ref_9fa8c7d6e5f4b3a2c1d0e9f8a7b6c5d4",
  "expiresIn": 900
}
```
* **Error Response (401 Unauthorized)**:
```json
{
  "errorCode": "ERR_INVALID_CREDENTIALS",
  "message": "Invalid email or password."
}
```

#### 28.1.3 POST /v1/auth/logout
* **Purpose**: Invalidates the active refresh token.
* **Authentication Required**: Yes.
* **Request Header**: `Authorization: Bearer <accessToken>`
* **Request Body**:
```json
{
  "refreshToken": "ref_9fa8c7d6e5f4b3a2c1d0e9f8a7b6c5d4"
}
```
* **Success Response (204 No Content)**: Empty body.

#### 28.1.4 POST /v1/auth/refresh
* **Purpose**: Rotates token credentials.
* **Authentication Required**: No.
* **Request Body**:
```json
{
  "refreshToken": "ref_9fa8c7d6e5f4b3a2c1d0e9f8a7b6c5d4"
}
```
* **Success Response (200 OK)**: Returns new `accessToken` and rotated `refreshToken`.

#### 28.1.5 POST /v1/auth/forgot-password
* **Purpose**: Triggers a password recovery email.
* **Request Body**:
```json
{
  "email": "rohan@domain.com"
}
```
* **Success Response (202 Accepted)**:
```json
{
  "message": "If the email exists, a password reset link has been sent."
}
```

#### 28.1.6 POST /v1/auth/reset-password
* **Purpose**: Resets the password using a recovery token.
* **Request Body**:
```json
{
  "resetToken": "tok_3fa9b1e2c3d4",
  "newPassword": "NewSecurePassword123!"
}
```
* **Success Response (200 OK)**:
```json
{
  "message": "Password updated successfully."
}
```

#### 28.1.7 POST /v1/auth/verify-email
* **Purpose**: Confirms the user's email address.
* **Request Body**:
```json
{
  "verificationToken": "vtok_8c7b6a5a4b3c"
}
```
* **Success Response (200 OK)**: Message confirming email verification.

#### 28.1.8 POST /v1/auth/change-password
* **Purpose**: Updates the password from within an active session.
* **Authentication Required**: Yes.
* **Request Body**:
```json
{
  "currentPassword": "Password123!",
  "newPassword": "NewSecurePassword123!"
}
```
* **Success Response (200 OK)**: Message confirming password change.

#### 28.1.9 DELETE /v1/auth/account
* **Purpose**: Flags the user account for deletion.
* **Authentication Required**: Yes.
* **Success Response (202 Accepted)**: Message confirming deletion scheduled.

## 29. DevOps, Deployment & CI/CD

### 29.1 Repository Folder Structure

 ├── cmd/
 │    └── api/
 │         └── main.go                 -- Application entry point
 ├── internal/

 │    ├── sync/                        -- Vector clock resolution handlers
 │    ├── sandbox/                     -- Kubernetes sandbox compilers
 │    ├── models/                      -- Entity definitions matching DB schemas

### 29.2 Containerization (Dockerfile)
The services compile into lightweight Scratch containers:
```dockerfile
# Stage 1: Build binary
FROM golang:1.21-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-w -s" -o main ./cmd/api

# Stage 2: Execute build
FROM scratch
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
COPY --from=builder /app/main /main
EXPOSE 8080
ENTRYPOINT ["/main"]
```

### 29.3 Secrets Management
* **Production Configurations**:
  * Secrets are not written directly to environment configuration files.
  * Services retrieve credentials from **AWS Secrets Manager** or **HashiCorp Vault** using system IAM role assignments.
  * Local development maps values from a locked `.env` file excluded from version control systems.

### 29.4 CI/CD Pipeline (GitHub Actions)
```yaml
name: Production CI/CD Pipeline

on:
  push:
    branches: [ main ]

jobs:
  test-and-lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Initialize Go env
        uses: actions/setup-go@v4
        with:
          go-version: '1.21'
      - name: Run code linter
        run: golangci-lint run ./...
      - name: Run tests
        run: go test -v -race ./...

  build-and-deploy:
    needs: test-and-lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v2
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1
      - name: Push Docker image to Amazon ECR
        run: |
          aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin 123456789.dkr.ecr.us-east-1.amazonaws.com
          docker build -t atharvaos-api:latest -f deployments/Dockerfile .
          docker tag atharvaos-api:latest 123456789.dkr.ecr.us-east-1.amazonaws.com/atharvaos-api:latest
          docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/atharvaos-api:latest
      - name: Rollout Blue-Green Deploy to Kubernetes
        run: |
          aws eks update-kubeconfig --name atharvaos-prod-cluster --region us-east-1
          kubectl apply -f deployments/k8s-deployment.yml
          kubectl rollout status deployment/atharvaos-api-deployment
```
